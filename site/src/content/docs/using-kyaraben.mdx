## The system picker

The main screen shows all available systems. Check a system to enable it,
uncheck to disable. Each system shows its available emulators. Click apply to
install or remove emulators based on your selection.

{/* Screenshot: v0.1-using-system-picker.png - System picker with emulator details */}

## What happens when you apply

When you click apply, kyaraben:

1. Shows a preflight check with the exact changes it will make
2. Fetches emulator packages (from a binary cache, so downloads are fast)
3. Installs emulator binaries to kyaraben-specific directories
4. Generates or updates emulator configuration files
5. Creates the folder structure in your `UserStore`
6. Creates desktop launchers for each emulator
7. Updates the manifest tracking what was installed

If apply fails partway through, the previous state remains intact. The manifest
is only updated after all steps succeed.

## BIOS and firmware files

Some systems require BIOS or firmware files to function. Kyaraben does not
distribute these files. It verifies them by hash to confirm you have the right
ones.

Use the provisions panel in the UI or run `kyaraben doctor` to see which files
are needed and whether they are found and verified.

{/* Screenshot: v0.1-using-doctor-view.png - Doctor view showing BIOS status */}

## Observability

Kyaraben tracks what it does and shows you the details.

### Preflight diffs

Before applying changes, kyaraben shows exactly which config files will be
created or modified, including line-by-line diffs:

```
Config changes:

  MODIFY ~/.config/retroarch/retroarch.cfg
    ~ system_directory
        - "/old/path/bios"
        + "/home/user/Emulation/bios"

  CREATE ~/.config/duckstation/settings.ini
    + [BIOS]SearchDirectory = "/home/user/Emulation/bios/psx"
    + [MemoryCards]Directory = "/home/user/Emulation/saves/psx"

  Summary: 1 file(s) to create, 1 to modify
```

### Status

`kyaraben status` shows the current state at a glance: which emulators are
installed, which config files are managed, and whether any user modifications
were detected.

{/* Screenshot: v0.1-using-status-output.png - Status command output */}

### What kyaraben installs and where

Kyaraben installs emulators and tools (like frontends) into kyaraben-specific
directories under `~/.local/share/kyaraben/`. Desktop launchers go to
`~/.local/share/applications/kyaraben/`. This means kyaraben does not overwrite
emulators you installed from other sources.

Emulators themselves may create additional files in their default config
locations (shader caches, input profiles, logs). Kyaraben does not manage
these files.

## Managed config keys

Kyaraben manages specific keys in emulator configuration files. These are
typically path settings that point emulators to your `UserStore`:

- ROM directories
- BIOS search paths
- Save file directories
- Save state directories
- Screenshot directories

Kyaraben uses the emulators' default config file locations. For example,
RetroArch configuration lives at `~/.config/retroarch/retroarch.cfg`, and
DuckStation at `~/.config/duckstation/settings.ini`.

### What kyaraben manages vs. what you control

Kyaraben only manages keys it needs to set for the `UserStore` integration.
All other settings (resolution, shaders, audio, display options) are yours to
configure freely in each emulator.

When you apply, kyaraben uses a three-way merge:

1. It compares what it last wrote (the baseline) with what is currently on disk
2. If you changed a key that kyaraben does not manage, your change is preserved
3. If you changed a managed key, kyaraben warns you before overwriting

This means you can customize emulator settings through their own UIs without
losing changes when you run apply again.

### Kyaraben as the emulator config manager

Kyaraben expects to be the sole manager of emulator configuration files. If
you use another tool (like EmuDeck) that also writes to the same config files,
the two will conflict.

You can install the same emulators separately from other sources. Since
kyaraben installs its own copies to kyaraben-specific directories, there is no
binary conflict. However, they will share the same config files at default
paths, and kyaraben will overwrite managed keys on apply.

## The `UserStore`

Your `UserStore` (default `~/Emulation`) has a consistent structure:

```
~/Emulation/
├── roms/              # Your ROM files, organized by system
│   ├── snes/
│   ├── psx/
│   └── ...
├── bios/              # BIOS and firmware files, organized by system
│   ├── psx/
│   └── ...
├── saves/             # Save files (memory cards, battery saves)
│   └── (per-system)
├── states/            # Emulator save states
│   └── (per-emulator)
├── screenshots/       # Captured screenshots
│   └── (per-system)
└── opaque/            # Emulators with complex internal structures
    ├── eden/          # Switch emulator data
    ├── rpcs3/         # PS3 emulator data
    └── ...
```

Some emulators have complex internal directory structures that do not fit the
`saves/states` model. These use an `opaque/` directory where the emulator
manages its own layout. Kyaraben points the emulator at this directory and
syncs it as a unit. See [how it works](/how-it-works/) for details.
